寻找拼图游戏最优解(8 Puzzle)

发表于 2018-06-11 | 分类于 作业报告

首先说明这是优先队列的编程作业。
题目给定一个n*n大小的数字板，里面随机分布着1~n*n-1 的n-1个数字，数字不会重复出现，板子中有一个格子是空的，与它相邻实心格子可以通过和它交换位置实现垂直和水平移动，要求我们把这块板子按照格子的布局从上到下，从左到右进行排序，空格子要放在右下角。
说了一大堆估计没人看，没图说个鸡儿。下图就是我们要做的事了。

刚看到这个图片时我陷入了沉思，这种排列的结构我好像在我爸的按键手机里见过，还伴随了我好一段无聊的时光。。。

仔细想想，这不就是特么的老式手机上的自带的出厂游戏，滑块拼图吗！

只要我把数字板上的数字按照编号一一对应到这个拼图上，输入给程序，然后按照程序最终给出的答案执行对应的流程，这个游戏不用动脑子就能玩了！终于实现了我多年来想学以致用的梦想，居然还是在作业里意外发现的，激动地我放弃了备考还有十天就要到来的六级，用了几天把代码实现了出来。

实现思路

总体思路

解决这个问题用A*算法作为核心思路，用优先队列作为实现这个思路的工具。我们假设的每一步操作后的数字板（拼图）都是一个节点，这个节点保存着它已经操作的次数和对实现目标预估的步数还有它的前趋节点。当最初的数字板传进来之后，将它的邻居节点（此节点移动的所有可能性）插入到优先队列（注意是MinPQ），然后delMin()一个优先级最低的出来，再将它的邻居节点插入到优先队列，直到delMin()出来最终想要的结果。

优先级

这个优先级是已经操作的步数+对实现目标预估的步数，那么怎么才能正确预估距离目标的步数呢，很抱歉，我想不出来完美的答案，而且好像发明这个算法的人也没想出完美的答案，我们只能作近似的预估，这种近似的预估有两种方法。

  1. Hamming：不在目标位置的格子的个数（除了空格子）
  2. Manhattan：每个不在目标位置格子与目标位置的距离之和（除了空格子）


请忽略图中的 “+ 0”。
题目给出了这两种方法，说正确地选择这两种优先级算法是这种思想能否成功的关键，并且在最后提交作业说明中说在实现中要使用Manhattan距离，这说明Hamming距离是不会被用到的，但API中却要求实现这个方法。开始我有点困惑，于是去checklist里看了一下代码优化方案，才知道Hamming距离可以在两个数字板的Manhattan距离相等的情况下比较，Hamming大的优先级就大，这算是一个小小的优化。

那Hamming距离究竟是什么原因不被使用呢？
这是最终版本的作业，使用Manhattan距离的时间空间占用测试和总分

Correctness: 49/49 tests passed
Memory: 22/22 tests passed
Timing: 125/125 tests passed

这是使用Hamming距离时候的情况

Correctness: 43/49 tests passed
Memory: 10/22 tests passed
Timing: 0/125 tests passed

内存时间都不合格，对于需要操作步数较多的情况，还会这样

OperationCountLimitExceededException
Number of calls to methods in MinPQ exceeds limit: 10000000

这说明了虽然做不到完美的预估，但是预估与实际的切合度要尽量高，否则非常影响算法效率。

前趋节点

到这里，我还没有说保存在每一个节点里前趋节点的作用，这是为了保证一个节点拓展出邻居的时候，防止重复添加它的上一个节点，这种问题文字描述起来很拗口，用图就会很好理解。
把前趋节点保存下来后，我们可以在添加邻居节点之前判断这个邻居是不是他的前趋节点而决定要不要放弃添加它，这不是优化那么简单，这是非常重要的一步操作，没有这一步很可能会导致内存崩溃。

判断题目是否无解

存在很多无解的数字板，比如
关于这种无解的板子，有一个神奇的现象：将一个有解的板子的任意两个非空元素交换，这个板子就成了无解板。根据这个事实很容易推出:将一个无解板的任意两个元素交换就能得到一个有解板。根据这个性质，在求解一块板子的同时，我们可以求解它的双胞胎（任意交换元素的板子），当它的双胞胎delMin()出答案时，就证明它是无解的。

实现

board类

board类的大部分方法没什么意思，而且代码很长，为了避免凑字数的嫌疑，我只选出比较有意义的贴出来。

“预算”

注意，这里“预算”的意思不是budget的那个预算，而是pre-compute的意思，真正的字面意义。
很多时候在构造器里，我们就能比较方便的算出其它方法需要的东西，在board中，我们将构造器的参数复制到这个类的域的时候，就可以乘机把dimension，manhattan，hamming算出来了，将它们保存到私有域里，在方法调用的时候只用调用私有域，而不用再次计算了。另一个角度说，这么做是用空间省时间，在这里用了常数的空间就换到了线性级别的时间，这波不亏哈哈。

有人会问为什么不在这把twin也一并算出来，原因有两点：

  1. 双胞胎是一个int[dimension][dimension]的数组,如果把双胞胎保存在私有域里的话，占用的不是常数的内存，而是dimension*dimension的内存，保存在私有域里挺不划算的。
  2. 即使保存在私有域里了，每次调用twin()函数的时候还是要再新建一个数组返回，因为要保证返回不可变，详情请看我的上上上篇博客（查找共线点）。

equals方法

为了这个方法，我专门开了篇博客讲它（基于类的访问权限)，但是那片博客还是没说完，因为我到最后交作业的时候检查器又让我改了。。。因为我开始想的是equals方法的参数类型应该和调用者的类型一致，所以我是这样声明方法的equals(Board y)，如果检测到传入参数不是Board类，抛出一个参数错误异常。提交上去后

fails to return false when argument is of type String

原来传入String进来不是要抛出异常而是返回false，我感觉很奇怪，搜索了一下，发现别人说如果equals()方法的参数不是object类型的话就不叫重写，而叫重载。意味着这不是修改了祖先类Object的equals方法，而是直接继承了下来，然后equals方法只是对它的重载，意味着equals(Object y)和equals(Board y)都能调用，并且是两个不同方法。
修改了之后果然好了，但我还是不理解为什么要这么做？为什么会有人比较两个不同类型的对象，这难道不是应该定义为不支持的操作吗。可能这是个代码规范吧，这样做在和父类的比较时可以少写一个方法。

toString方法

题目要求toString方法打印出这个样式

开始的版本中，我先创建了一个String s变量，然后写了很多的s+=开头的句子拼接字符串，然后

M P SBSC_USE_STRINGBUFFER_CONCATENATION SBSC: Builds a ‘String’ object using the ‘+’ operator in a loop, which can take time quadratic in the length of the resulting string. Instead, use ‘StringBuilder’. At Board.java:[line 186]

其实我以前是大概知道StringBuilder这个类的，也隐约知道它是可变的字符串，但是印象不大，更不知道使用场景，借这个机会温习一下这个类的特点。Sting类型是不可变字符串，所以看上去好像是在做拼接操作的s+=实际上是新建了一个字符串，把原字符串里的内容拷贝进去再加上“=”后面的内容。这样做的n次就会有n2/2的运行时间。而StringBuilder是可变字符串，每次调用append()方法是真的在拼接，这样调用n次就只有和n成正比的时间。
所以在需要大量拼接的场景中，用StringBuilder比String更省时间空间。

Node

如何实现思路里的节点的问题。因为作业要求里说明了只能提交board.java和solver.java两个文件，而且课程里好像也没讲到过内部类（虽然我知道这个东西），所以我没想过为Node单独创建一个类，但是如何保存一个节点已经走过的步数和它的前趋节点呢？我一开始准备在Board中定义，但是又要遵守API，所以不能有其它public的变量和方法。在Solver里定义也是不行，想了好久好久都没有答案。在讨论里搜了一下才知道大家都是创建内部类的，果然太保守害死人啊，如果我早一点知道可以使用内部类估计能省不少时间。。

域的空间>方法的空间

除了优先队列minPQ和twinMinPQ之外，我还用到了两个栈，一个stack是构造器方法里的变量，另一个boardStack是私有域里的变量。在构造器中，minPQ每delMin()出来一个节点，stack就会将此节点里面的的board添加进栈里。这个栈的轨迹不能直接作为解拼图的方法轨迹，因为里面包含一些尝试过但是最终放弃的路径。80-92行将stack进行了一次加工放入了boardStack里。客户代码每次调用solution()的时候，程序会再创建一个queue，把boardStack里的内容复制到queue里之后返回queue。
开始的时候我不是这样做的，我只在私有域里用了一个stack存放原始的那个栈，每次调用solution()的时候再创建一个tmpStack，把stack加工一下复制进tmpStack，最后返回tmpStack。
然后我发现在我提交之后，无论如何都无法拿到满分，总是会有4个内存测试不能通过

Correctness: 49/49 tests passed
Memory: 18/22 tests passed
Timing: 125/125 tests passed
Aggregate score: 98.18%

这种是最烦的，因为它就告诉你内存用多了，也不告诉你哪里出了错，每次看到99/100这个分数我的强迫症就会犯，检查了好几次代码都感觉没问题。然后就又跑到讨论里搜，很难受的是这种问题也不好搜，于是我的用的关键字就是“memory”，“99/100”，“98/100”这种，显然很难定位我的问题。但我看到其中有一个人说了这个

I have found the problem. If I put the priority queue inside the constructor ( Solver(Board initial) method) , I can pass the memory test. However, if I define the priority queue as a private field of the Solver class, the memory problem shows up.

这个的确是我犯过的一个问题，不过是我早就把它改掉了，但他给了我一点启发，于是我去检查了一下私有域，然后把代码改成了现在的样子，提交之后就看到了这种能让我分泌一个可乐瓶多巴胺那么多量的提示。
那么后一种方案比前一种好在哪呢？
首先，为什么要写成一开始那种形式，一是因为我不想声明那么多变量，这看起来很麻烦，二是我觉得这样做会比较省时间(省构造的时间)。
但是这样做会导致空间占用过多：方法的空间不值钱，因为每当一个方法结束了之后它声明变量所占用的空间就会被垃圾回收器收回。但是如果一个变量是在域里面的，只要客户代码还保留着这个对象的引用，这个对象里的变量就会一直存在。
在我的第一个版本中，直接把原始的轨迹保存在私有域里，这个原始轨迹掺着很多没用的东西，会占用过多的内存，而在构造器里就将它加工了一下再放入就会省下这些内存。
另外，最初版本也不见得省下多少时间，如果solution()调用了不止一次的话。
