题目要求

Given a set of n distinct points in the plane, find every (maximal) line segment that connects a subset of 4 or more of the points.

我来机翻一下：在一个平面中给定n个点，找到每个含有四个点或更多点的子集组成的线段。

要完成上面的要求，题目给出了两个数据结构，分别是Point和LineSegment。

后三个方法是要我们写的，篇幅有限，不能具体说明。细节要求请看specification

LineSegment 不用动它，这是已经实现好的
Point和LineSegment是点和线的表示方法，下面是解决问题的方法

暴力解法，这种解法会检查每一组四点线段，然后判断每是否共线，segments()会返回一个LineSegment数组，这个数组包含了每一个四点共线的线段，并且不允许重复，如果线段是A>B>C>D，那么数组只能包含A>D或者D>A，这两个线段其中之一，而不能包含子线段比如A>C,或者B>D。
性能要求:运行时间n4，内存使用与n成正比。

快速版本：这种解法可以分成四步

  1. 把p点看作初始点
  2. 对于每一个q，将他们按照与p点的斜率大小排序
  3. 遍历排序好的数组，如果连续的三个（或更多）q与p的斜率相同，则这三个点（或更多）与p就是共线的
  4. 对于每一个点，循环1，2，3步骤。


快速解法的API和暴力解法一样，不过它支持4个以上的点共线，细节要求请见specification
性能要求：时间要求为n2*log n，空间要求和n成正比

程序流程

代码的13到26行是检查输入参数是否有问题（数组不得为null，数组中不得含有null元素，不得含有重复元素）并按照从高到低排好序。28到45行是遍历各点的所有可能的组合（不是排列），并且用isCollinearPoints（）函数检查每一组点是否共线，如果共线，则用这一组点的 points[first] 和 points[fourth] ，new 一个 LineSegment,push到栈 lines 里，再将numberIndex ++。
也就是说，对象在构造的时候，程序就已经找到所有的共线点了，当用户调用segments()的时候，程序再用迭代器（而不是用pop()）将栈里的元素送进一个新的LineSegment数组中，并且返回这个数组。

debug之路

BruteCollinearPoints 和 FastCollinearPoints 犯了很多一样的错误，我都在这里说了，FastCollinearPoints 那里就不再说了。

递归还是循环

开始的时候我想用递归的方式遍历所有组合，顺便再强化一下使用递归的能力，写了好久终于把代码调试出来了，结果coursera检查我作业的时候用了好多Point，然后我的代码就StackOverflow了。。，于是我只能含泪把递归删了改用循环。（47行还有我用递归的痕迹）可能是我还不太会用递归，也可能这里本来就不适合递归，反正我想了好久都写不出来。。（好气）

不改实参

这是代码规范相关的错误，就是只能读实参，不能改实参。因为如果我们把实参改了，客户代码如果没有按照规范来写的话，说不定客户代码之后还要用到它，但是他们不知道我们其实是已经把实参内容改过了的，这样子就会出差错。之前有想过这个问题，但是没有人和我说过这个需要注意，我以为就不用在意这个，然后我提交上去之后作业检查器就给我了这个

Test 11: check that data type does not mutate the constructor argument

  • input8.txt
    • data type mutated the points[] array
    • data type should have no side effects unless documented in API
  • equidistant.txt
    • data type mutated the points[] array
    • data type should have no side effects unless documented in API
      ==> FAILED

偷懒被发现，只能老老实实再新建一个Point[ ] ，把实参复制进去。

返回值不可变

这个也是代码规范相关的问题，而且和上面那个是相爱相杀的关系，它们两个只要有一个遵守规范就不会出错，上面的是不能修改实参，而这个是直接把传递出去的值设置为不可变的，这里的不可变不是真的不可变，而是调用端相对于被调用端不可变。很不幸，在我最初的代码版本里，这两个规范我都没有遵守，这意味着我永远不能调用自己写的代码。。。幸运的是，我是个知错就改的好孩子，以后应该不会出现这种问题了。。。解决办法和上面也是对称的，创建一个副本，把原始数据拷进去，然后再输出。

事实上，在做完这个之后，我同时解决了另一个问题。
开始，我并没有用副本，segments()方法直接返回原数组，但是这就出现了一个问题，就是我不知道以后可能会有多少个共四点线段，又懒得写可调节数组的代码。于是我就在程序开始时计算参数中的点可能有多少中组合，按照最大值这样创建数组。这样做完后，又出现了两个问题

  1. 我们知道，从n个元素中任取m个元素组合，有C=n!/(m!*(n-m)!)种组合方式，这里m=4，代入C=n!/(4*3*2*(n-4)!)，约等于n4，这是个什么概念，我从测试文件夹中找到的一个文件里面有1260个点，12604=2520473760000，而int类型的最大值才为2^31-1 = 2147483647，创建这么大的数组根本不可能。
  2. 假设我又添加了可调节数组的代码，可是按照一般情况，数组是不会被填满的，可是作业检查器会用for each语法糖来遍历我的数组，这样就会出现NonPointException

被迫使用副本。意外的是，当我开始用副本后，感觉一切都明朗起来，尤其是第一个问题，用数组很棘手，既然最后省不掉副本，那么在收集共点线的时候就可以用其它的数据结构，比如栈！这样就可以不用关心长度了，push进栈的时候把numberIndex ++，最后创建副本的时候这样LineSegment[] tmp = new LineSegment[numberIndex]; perfect！

性能分析

暴力解法最大的问题就是要遍历每四个点的所有组合，前面已经说过了当输入点的个数为n的时候，共有n4种组合，所以这个解法的性能就是O（n4)，刚刚已经计算过四次方级别是多么恐怖。空间性能占用的不大，甚至没有达到正比于n，但是这么恐怖的时间复杂度，空间就算一点不占也么的用。

关于重复线段和子线段的问题

这个问题在暴力解决方案里不存在，因为暴力解决虽然遍历了一遍所有组合，但是它只遍历了一遍，所以不会出现重复线段的问题，而且暴力解决默认之解决只含有4个元素的问题，所以不会出现子线段问题。而快速解决方案的API里吹下了牛逼，说能解决4个以上共线点的问题，而且还不出现重复线段和子线段，这就很头疼了。开始我没有考虑这个问题，而是想着先把线段找出来，然后再判断lines里是否已包含了这个线段，如果没有再添加进去。而这个判断lines里是否已经包含这个线段不是说起来那么容易，因为LineSegment这个类里没有equals()或者compareTo()这两个方法，甚至hashCode()方法也被重写成了

所有比较两个线段是否相等的路都被封死了。但是机智如我，最后竟然发现LineSegment竟然重写了toString()，我只要比较两个线段的字符串形式不就行了吗，说干就干，最后代码也成功运行了，但是我用的时候忽略了toString()上面的注释

Returns a string representation of this line segment
This method is provide for debugging;
your program should not rely on the format of the string representation.

果然，在信心满满地提交完作业之后，分数依旧不理想，作业检查器说我的代码

fragile dependence on return value of toString()

一个月过去了，作业检查器一点没变，还是那么严格。。。然后我想能不能不用LineSegment里的代码检查重复，而是在每次保存的到lines 里的时候再保存一份其他的能表示唯一线段的数据到另一个地方，然后每次检查重复就用另一个地方的什么方法检查。但是感觉这种办法绕了好远，自己又想不出更好的解决办法，于是跑到讨论里搜索了一波。

甚至不用关键字搜索，仅仅点击用最热门方式排序，第一个就是我碰到的问题，看来我是正常人（￣︶￣）↗　，点进去看看：

这位兄dei的想法和我一样，可是我感觉这种方法实在太绕了，于是继续搜索

这个答案有三个赞！感觉有戏，于是耐着性子把这么长一段英文看完，终于找到了满意的解法！首先，按照和p[i]的斜率将所有点排序，然后寻找三个以上相邻的，有相同斜率的点，并且将它们用compareTo()（从高到低，从右到左）方法排序，然后用最小的那个和p[i]用compareTo()比较，如果p[i]比那个最小的还要小（意味着它是最低的或者最左边的），那么就用最大的那个点和p[i]组成一个线段 push 进去，这样就可以避免重复元素了！这样既节省代码又节省时间，出题者把其他的路封得那么死，就是为了引导我们用这样的方法吧。

性能分析

44行到62行是一个执行n次的外循环，里面又很多内循环，我们只用找出其中循环最多的那个内循环就行，其他的项忽略，经过分析，只有调用排序那一行代码的循环次数最多，也就是快速排序的效率n*log n，那么叠加起来就是n2*log n了，符合要求。空间上面使用了一个辅助数组，也就是与n成正比了，也符合要求，任务完成！
